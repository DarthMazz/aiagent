# ルール

このファイルはAIエージェントの動作に関するルールとガイドラインを定義します。

## Python開発基本ルール

1. **Pythonバージョン**: Python 3.12以上を対象とする
2. **PEP 8準拠**: Pythonコーディング規約（PEP 8）を厳守する
3. **仮想環境**: プロジェクトごとに仮想環境（venv）を使用する
4. **UTF-8エンコーディング**: すべてのソースファイルはUTF-8で保存する

## コード品質ルール

1. **最小限の変更**: 目的を達成するために最小限の変更を行う
2. **既存機能の保持**: 動作中のコードを壊さない
3. **規約の遵守**: 既存のコードスタイルとパターンを尊重する
4. **コミット前のテスト**: 変更がテストを壊さないことを確認する

### Python特有のコード品質ルール

1. **型ヒント**: 可能な限り型ヒント（Type Hints）を使用する
   ```python
   def process_data(items: list[str]) -> dict[str, int]:
       ...
   ```

2. **Docstring**: すべての関数、クラス、モジュールにdocstringを記述する
   - 関数: 引数、戻り値、例外を明記
   - クラス: 目的と主要なメソッドを説明

3. **エラーハンドリング**: 適切な例外処理を実装する
   - 具体的な例外をキャッチする（bare `except:` は避ける）
   - 必要に応じてカスタム例外を定義する

4. **モジュール構成**: 適切なファイル・ディレクトリ構造を維持する
   ```
   project/
   ├── src/
   │   └── package_name/
   │       ├── __init__.py
   │       └── module.py
   ├── tests/
   ├── requirements.txt
   └── README.md
   ```

## Linter・フォーマッタールール

### Ruffの使用

1. **Linterツール**: コードの静的解析には `ruff` を使用する
   - 高速で包括的なPython linter
   - Flake8、isort、pyupgradeなどの機能を統合

2. **Lintチェック**: コード変更後は必ず `uv run ruff check .` を実行する
   - コミット前に全てのlintエラーを修正する
   - 自動修正可能なものは `uv run ruff check --fix .` で修正

3. **フォーマッター**: コードフォーマットにも `ruff` を使用する
   - `uv run ruff format .` でコードを自動フォーマット
   - Black互換のフォーマット規則

4. **設定ファイル**: `pyproject.toml` でruffの設定を管理する
   ```toml
   [tool.ruff]
   line-length = 88
   target-version = "py312"
   ```

5. **CI統合**: 継続的インテグレーションでlintチェックを自動実行する

## 依存関係管理ルール

### パッケージマネージャー

1. **uvの使用**: パッケージ管理には `uv` を使用する
   - pipの代わりに `uv` を使用する
   - 高速なパッケージインストールと依存関係解決

### 依存関係ファイル

1. **pyproject.toml**: プロジェクトの依存関係を管理する
   - `uv add <package>` でパッケージを追加
   - `uv remove <package>` でパッケージを削除

2. **uv.lock**: 依存関係のロックファイルを維持する
   - バージョンの固定と再現性を保証
   - このファイルはコミットする

3. **バージョン管理**: セキュリティ上重要な依存関係のバージョンを明示する

### uvコマンド規則

1. **仮想環境作成**: `uv venv` で仮想環境を作成する

2. **パッケージインストール**: `uv sync` で依存関係をインストールする
   - 新規クローン後は `uv sync` を実行
   - 依存関係変更後も `uv sync` を実行

3. **パッケージ追加**: `uv add <package>` で新しい依存関係を追加する
   - 自動的に pyproject.toml と uv.lock を更新

4. **開発用依存関係**: `uv add --dev <package>` で開発用パッケージを追加する

5. **パッケージ実行**: `uv run <command>` でプロジェクトのコンテキストでコマンドを実行する

## ドメイン駆動設計（DDD）ルール

### 戦略的設計ルール

1. **ユビキタス言語**: チーム全体で統一された用語を使用する
   - ドメインエキスパートと同じ用語をコードに反映
   - クラス名、メソッド名、変数名にビジネス用語を使用
   - 用語集を維持・更新する

2. **境界づけられたコンテキスト**: ドメインを明確なコンテキストに分割する
   - 各コンテキストは独自のモデルを持つ
   - コンテキスト間の境界を明確にする
   - コンテキストごとにディレクトリを分離

3. **コンテキストマップ**: コンテキスト間の関係を文書化する
   - 上流/下流の関係を明確化
   - 統合パターン（共有カーネル、順応者など）を定義

### 戦術的設計ルール

1. **エンティティ**: アイデンティティを持つオブジェクト
   - 一意のIDを持つ
   - ライフサイクルを通じて追跡可能
   - 同一性は属性ではなくIDで判断

2. **値オブジェクト**: 不変で属性で識別されるオブジェクト
   - 不変性を保証する（`@dataclass(frozen=True)` を使用）
   - 等価性は全属性の比較で判断
   - 副作用のないメソッドのみ持つ

3. **集約**: 一貫性境界を持つエンティティと値オブジェクトの集合
   - 集約ルートを通じてのみ外部からアクセス
   - 集約内の一貫性をトランザクション境界で保証
   - 集約は小さく保つ（必要最小限の範囲）
   - 集約間の参照はIDで行う

4. **リポジトリ**: 集約の永続化を抽象化
   - 集約ごとにリポジトリを定義
   - インターフェース（Protocol/ABC）で定義
   - コレクションのように振る舞う

5. **ドメインサービス**: エンティティや値オブジェクトに属さないドメインロジック
   - ステートレスな操作
   - ドメイン知識を表現
   - 複数の集約にまたがるロジック

6. **ドメインイベント**: ドメインで発生した重要な出来事
   - 過去形で命名（`OrderPlaced`, `PaymentProcessed`）
   - 不変オブジェクトとして実装
   - イベント駆動アーキテクチャの基礎

### ヘキサゴナルアーキテクチャルール

1. **レイヤー構成**: 以下の構造を維持する
   ```
   project/
   ├── domain/              # ドメイン層（コアビジネスロジック）
   │   ├── entities/        # エンティティ
   │   ├── value_objects/   # 値オブジェクト
   │   ├── repositories/    # リポジトリインターフェース
   │   ├── services/        # ドメインサービス
   │   └── events/          # ドメインイベント
   ├── application/         # アプリケーション層（ユースケース）
   │   ├── use_cases/       # ユースケース
   │   └── dto/             # データ転送オブジェクト
   ├── infrastructure/      # インフラ層（技術的実装）
   │   ├── persistence/     # データベース実装
   │   └── messaging/       # メッセージング実装
   └── adapters/            # アダプター層（外部接続）
       ├── api/             # REST API、GraphQL等
       ├── cli/             # コマンドラインインターフェース
       └── ui/              # ユーザーインターフェース
   ```

2. **依存関係の方向**: 依存性逆転の原則（DIP）を適用
   - ドメイン層は他の層に依存しない（完全に独立）
   - アプリケーション層はドメイン層のみに依存
   - インフラ層とアダプター層はポート（インターフェース）を実装
   - 依存関係は常に内側（ドメイン）に向かう

3. **ポートとアダプター**: インターフェースと実装を分離
   - ポート: ドメイン/アプリケーション層でインターフェース定義
   - アダプター: インフラ層で具体的な実装
   - 技術的詳細をドメインから隔離

### 命名規則

1. **エンティティ**: ビジネス用語の名詞（`User`, `Order`, `Product`）
2. **値オブジェクト**: 概念を表す名詞（`Email`, `Money`, `Address`, `OrderStatus`）
3. **集約ルート**: ビジネス上の主要な概念（`Order`, `Customer`）
4. **リポジトリ**: `{エンティティ名}Repository`（`UserRepository`, `OrderRepository`）
5. **ドメインサービス**: `{動作}Service`（`PricingService`, `ShippingCalculator`）
6. **ユースケース**: `{動詞}{名詞}UseCase`（`CreateOrderUseCase`, `ProcessPaymentUseCase`）
7. **ドメインイベント**: 過去形（`OrderPlaced`, `PaymentProcessed`, `UserRegistered`）

### 実装ガイドライン

1. **型ヒントの強化**: Protocol、ABC、Generic を活用
2. **不変性の保証**: `@dataclass(frozen=True)` や `Pydantic` を活用
3. **ドメインロジックの保護**: ドメイン層に技術的詳細を持ち込まない
4. **テスタビリティ**: ポート/アダプターパターンでモック可能に

## テストルール

1. **pytest使用**: テストフレームワークとしてpytestを推奨する

2. **テストファイル命名**: `test_*.py` または `*_test.py` の形式を使用する

3. **テストカバレッジ**: 主要な機能には必ずテストを書く

4. **テスト実行**: コード変更後は関連テストを実行して動作確認する

## コミュニケーションルール

1. **簡潔さ**: 回答は簡潔に要点を絞る
2. **不明点は質問**: 推測ではなく明確化のための質問をする
3. **行動の説明**: 重要な決定には文脈を提供する
4. **進捗報告**: 長時間の操作中はステータスを更新する

## セキュリティルール

1. **機密情報の除外**: 認証情報や機密データをコミットしない
2. **入力検証**: コマンド実行前にユーザー入力を検証する
3. **重要データのバックアップ**: 変更時には既存ファイルを保持する
4. **安全なコマンド**: 確認なしに破壊的操作を行わない

## ワークフロールール

1. **事前計画**: 複雑なタスクは実装前に計画を作成する
2. **変更の検証**: タスク完了前に変更をレビューしてテストする
3. **クリーンアップ**: 完了後は一時ファイルとリソースを削除する
4. **決定の文書化**: 自明でない実装選択は説明する

## AI駆動開発ライフサイクル（AIDLC）ルール

AIとの協働による開発ライフサイクルの各フェーズで守るべきルールを定義する。

### フェーズ1: 要件定義

1. **要件の明文化**: 開発開始前にAIとの対話で要件を整理・文書化する
   - ユーザーストーリー形式（`〜として、〜したい、なぜなら〜`）で記述
   - 受け入れ条件（Acceptance Criteria）を明確にする
   - 曖昧な要件は実装前に必ず確認する

2. **スコープの確定**: 実装範囲を明確に定義する
   - 対象と対象外を明示する
   - 段階的な実装計画を立てる

### フェーズ2: 設計

1. **設計先行**: 実装前に設計案をAIに提示してレビューを受ける
   - DDDのコンテキストマップを先に定義する
   - ヘキサゴナルアーキテクチャのレイヤー構成を確認する

2. **ADRの記録**: 重要なアーキテクチャ決定はADRに記録する
   - 決定の背景・理由・トレードオフを明記
   - `docs/adr/` ディレクトリに格納する

3. **設計レビュー**: 複数の設計案を比較してトレードオフを評価する

### フェーズ3: 実装

1. **TDDの実践**: テストを先に書いてからコードを実装する
   - Red（失敗するテスト）→ Green（テストを通す）→ Refactor（改善）

2. **小さな単位**: 小さいコード単位でAIと協働・確認しながら実装する
   - 一度に大量のコードを生成させない
   - 各ステップで動作確認を行う

3. **コードレビュー**: AIが生成したコードは必ず人間がレビューしてから採用する
   - ビジネスロジックの正確性を確認
   - セキュリティ・パフォーマンスの観点でチェック

### フェーズ4: テスト・検証

1. **テストの義務化**: AIが生成したコードには必ずテストを書く
   - ユニットテストでドメインロジックを検証
   - インテグレーションテストでレイヤー間の結合を検証

2. **Lintの通過**: コミット前に `uv run ruff check .` を実行してエラーをゼロにする

3. **動作確認**: テストをパスした後にコミットする

### フェーズ5: リファクタリング

1. **継続的改善**: 定期的にAIとコードレビューを実施して技術的負債を解消する

2. **モデルの深化**: ドメインの理解が深まったらコードに反映する
   - ユビキタス言語の更新
   - 集約境界の見直し

3. **テストの維持**: リファクタリング後も全テストがパスすることを確認する

### フェーズ6: 振り返り・改善

1. **パターンの蓄積**: うまくいった開発パターンをSKILLSに追加する

2. **ルールの更新**: 開発を通じて得た知見をRULESに反映する

3. **プロンプトの改善**: AIへの指示方法を継続的に改善する
   - 良い結果が得られたプロンプトを記録
   - 失敗パターンを記録して再発を防止
